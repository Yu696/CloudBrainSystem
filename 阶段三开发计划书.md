# 云脑诊疗平台 — 阶段三开发计划书

> 依据《云脑诊疗平台-微服务架构方案.md》阶段三路线图  
> 模块：**M05 影像管理 + M07 药库管理**

---

## 一、阶段三概述

| 项目 | 说明 |
|------|------|
| **阶段目标** | 补全影像管理（M05）和药库管理（M07）两大模块，涵盖 18 个 API + 9 个前端页面 |
| **对应模块** | M05 影像管理（DICOM 上传/预览/标注/对比/格式转换） + M07 药库管理（药品 CRUD/库存/发药/取药单） |
| **架构模式** | 单体架构内新增影像服务和药库服务层 |
| **数据库** | 已有表（M05 4 张 + M07 5 张，SQL 已就绪），补充 Drug 实体缺失字段 + 种子数据 |
| **预计工作量** | 后端 18 个 API + 前端 9 个页面 |

---

## 二、功能列表

### 2.1 M05 影像管理

| 功能编号 | 功能名称 | 说明 | 优先级 |
|---------|---------|------|--------|
| IM-01 | 影像上传 | 支持 DICOM/JPG/PNG 文件上传，存储至 MinIO/本地 | P0 |
| IM-02 | 批量上传 | 批量上传多张影像 | P1 |
| IM-03 | 影像信息查询 | 按患者/检查单查询影像列表和详情；**医生默认查看自己开单的影像** | P0 |
| IM-04 | 影像预览 | 浏览器端渲染影像（DICOM → PNG 预览） | P0 |
| IM-05 | 影像删除 | 逻辑删除影像文件 | P1 |
| IM-06 | 影像标注 | 病灶区域标注（矩形/圆形/文字），标注数据 JSON 存储 | P1 |
| IM-07 | 影像参数调整 | 亮度/对比度/缩放（前端处理为主） | P2 |
| IM-08 | 影像对比 | 两张影像并排对比 | P2 |
| IM-09 | 格式转换 | DICOM ↔ JPG/PNG 转换，记录转换日志 | P2 |

> **关于「检查医生」（影像科医师/Radiologist）角色**：业务上真实合理（跨科室阅片 + 出具影像初诊），但阶段三不做独立角色。阶段三影像 API 统一由现有 `doctor` 角色调用，数据模型预留 `uploader_id`（已有）和 `crossDept` 查询参数作为后续扩展点。阶段四或微服务拆分时再加 `ROLE_RADIOLOGIST`，改动集中在权限层，Service 层不动。

### 2.2 M07 药库管理

| 功能编号 | 功能名称 | 说明 | 优先级 |
|---------|---------|------|--------|
| DR-01 | 药品录入 | 新增药品信息（名称/编码/成分/规格/剂型/价格） | P0 |
| DR-02 | 药品修改 | 更新药品信息 | P0 |
| DR-03 | 药品删除 | 停用药品（逻辑删除） | P0 |
| DR-04 | 药品搜索 | 按名称/编码/分类模糊搜索，分页列表 | P0 |
| DR-05 | 库存查询 | 查看药品实时库存，含批次/有效期 | P0 |
| DR-06 | 库存预警 | 低库存/近效期/积压预警列表 | P1 |
| DR-07 | 发药出库 | 关联处方发药，扣减库存，生成出药记录 | P0 |
| DR-08 | 取药单打印 | 生成取药单，支持打印 | P1 |
| DR-09 | 仓库管理 | 药库/药房仓库的增删改查 | P2 |

#### 3.0.3 通用响应格式

所有 API 统一使用项目已有的 `Result<T>` 封装：

```json
// 成功
{ "code": 200, "message": "success", "data": { ... } }

// 业务异常
{ "code": 400, "message": "药品编码已存在", "data": null }

// 文件上传失败
{ "code": 500, "message": "影像文件格式不支持", "data": null }
```

#### 3.0.4 鉴权与数据可见性

| API 分组 | 所需角色 | 数据可见性 | Token 位置 |
|---------|---------|-----------|-----------|
| 影像管理 API | `userType=0`（医生） | 医生仅见自己开单的影像；管理员可见全部 | Header `Authorization: Bearer <token>` |
| 药库管理 API（全部） | `userType=1`（管理员） | 全量 | 同上 |

### 3.1 影像上传（IM-01）

```
POST /api/image/upload
Content-Type: multipart/form-data
```

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `file` | File | 是 | 影像文件（最大 50MB） |
| `patientId` | String | 是 | 患者 ID |
| `examinationId` | String | 否 | 关联检查单 ID |
| `modality` | String | 否 | 成像设备（CT/MRI/X光/超声） |
| `bodyPart` | String | 否 | 身体部位 |

**响应：**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "imageId": "IMG_a1b2c3d4e5f6g7h8",
    "imageName": "CT_chest_20260625.dcm",
    "imageType": 0,
    "imageTypeName": "DICOM",
    "fileSize": 15728640,
    "modality": "CT",
    "bodyPart": "胸部",
    "uploadTime": "2026-06-25 14:30:00"
  }
}
```

### 3.2 影像列表查询（IM-03）

```
GET /api/image/list?patientId=PAT_001&page=1&pageSize=10
```

**鉴权行为：**
- **医生角色**：默认只返回当前医生开过的检查单（`examination_order.doctor_id = 当前用户`）关联的影像
- **管理员角色**：可查看全部影像（不受 doctor_id 限制）
- 预留 `crossDept` 参数（阶段三暂不启用），后续 radiologist 角色通过此参数跨全科室查询

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `patientId` | String | 否 | 按患者筛选 |
| `examinationId` | String | 否 | 按检查单筛选 |
| `modality` | String | 否 | 按设备筛选 |
| `myExams` | boolean | 否 | 仅看自己开单的影像（医生默认 true，管理员默认 false） |
| `page` | int | 否 | 页码，默认 1 |
| `pageSize` | int | 否 | 每页条数，默认 10 |

**响应：**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "total": 15,
    "page": 1,
    "pageSize": 10,
    "records": [
      {
        "imageId": "IMG_a1b2c3d4",
        "imageName": "CT_chest_20260625.dcm",
        "imageType": 0,
        "imageTypeName": "DICOM",
        "modality": "CT",
        "bodyPart": "胸部",
        "fileSize": 15728640,
        "width": 512,
        "height": 512,
        "status": 1,
        "uploadTime": "2026-06-25 14:30:00"
      }
    ]
  }
}
```

### 3.8 发药出库（DR-07）

```
POST /api/drug/dispense
```

**请求体：**
```json
{
  "prescriptionId": "PRS_a1b2c3d4",
  "patientId": "PAT_001",
  "items": [
    { "drugId": "DRUG_001", "quantity": 2 },
    { "drugId": "DRUG_002", "quantity": 1 }
  ]
}
```

**响应：**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "recordId": "SHP_a1b2c3d4e5",
    "prescriptionId": "PRS_a1b2c3d4",
    "shipNum": 3,
    "payAmount": 55.50,
    "shipTime": "2026-06-25 16:00:00",
    "printStatus": 0
  }
}
```

**业务规则：**
- 校验每种药品库存是否充足
- 扣减库存（事务性）
- 若发药后低于 `min_stock`，自动生成库存预警记录
- 若药品已过期（`expiry_date < now`），拒绝发药

**设计决策：**
- `StorageService` 接口定义 `store()`、`retrieve()`、`delete()` 三个核心方法
- 阶段三默认使用**本地文件系统实现**，路径配置在 `application.yml`
- 预留 MinIO 实现接口，未来切换只需改配置
- 上传文件存储路径：`{storage.base-path}/{yyyy/MM/dd}/{imageId}.{ext}`
- DICOM 文件同时生成 PNG 缩略图用于预览

### 5.3 药库发药流程

```
处方提交 → 管理员审核 → 选择发药
                         │
                         ├─ 校验库存充足（事务开始）
                         ├─ 扣减 drug_stock.current_stock
                         ├─ 生成 ship_record 记录
                         ├─ 更新 prescription.status → 已发药
                         ├─ 检查库存是否低于 min_stock → 自动生成 stock_alert
                         └─ 返回发药结果（事务提交）
```

**关键规则：**
- 发药扣库存使用 `@Transactional` + 数据库行锁防止超卖
- 过期药品自动拒绝：`expiry_date < now` → `BusinessException("药品已过期，无法发药")`
- 库存不足拒绝：`currentStock < quantity` → `BusinessException("库存不足")`
- 发药后自动检查预警线

## 十一、安全与工程化设计

### 11.1 文件上传安全

| 措施 | 说明 |
|------|------|
| 文件类型校验 | 后端校验文件魔数（DICOM 文件头 128 字节偏移处 "DICM" 标识），非白名单格式拒绝 |
| 文件大小限制 | 单文件 50MB，批量上传 200MB |
| 路径遍历防护 | 文件名使用 UUID 重命名，禁止保留原始文件名 |
| 病毒扫描 | 阶段三暂不引入，阶段五（微服务）可扩展 ClamAV |

### 11.2 DICOM 格式处理策略

| 方案 | 复杂度 | 说明 |
|------|--------|------|
| **阶段三方案（推荐）** | 低 | 后端仅做 DICOM 格式识别 + 元信息存储，**前端**使用 `cornerstone.js` 进行浏览器端渲染。不做服务端 DICOM→PNG 转换 |
| 完整方案 | 高 | 引入 `dcm4che` 库做服务端 DICOM 解析和转换，阶段五微服务拆分时评估 |

**决策**：阶段三采用轻量方案 — DICOM 文件正常上传存储，影像预览由前端 `cornerstone.js` 库在浏览器端渲染。后端仅负责文件存储、元信息管理和标注数据存储。

### 11.3 发药防超卖

```
发药流程中的并发控制：
1. @Transactional 确保原子性
2. 扣减库存使用 UPDATE drug_stock SET current_stock = current_stock - ?
   WHERE drug_id = ? AND current_stock >= ? （乐观锁）
3. 若 affected rows = 0 → 库存不足，回滚事务
```

不引入 Redis 分布式锁（发药场景并发量低），使用数据库行级乐观锁即可。
